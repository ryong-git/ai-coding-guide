## ⚡ Quick Win #1: AWS 리소스 현황 파악 (10분)

### 왜 이걸 먼저?
- 읽기 전용이라 안전함
- 즉시 가치 있는 정보 획득
- 복잡한 설정 불필요

### 단계별 가이드

**1단계: 데이터 수집 (2분)**
```bash
# 현재 EC2 인스턴스 목록 가져오기
aws --profile dev-env ec2 describe-instances \
  --query 'Reservations[*].Instances[*].[InstanceId,InstanceType,State.Name,Tags[?Key==`Name`].Value|[0]]' \
  --output table > my-instances.txt
```

**2단계: Claude/Q에게 분석 요청 (1분)**
```
이 EC2 인스턴스 목록을 분석해서 다음을 알려줘:

[my-instances.txt 내용 붙여넣기]

1. 인스턴스 타입별 개수
2. 현재 실행 중인 인스턴스 수
3. Stopped 상태인 인스턴스 (비용 낭비 가능성)
4. 이름 태그가 없는 인스턴스
5. 간단한 최적화 제안
```

**3단계: 결과 확인 및 활용 (7분)**
- AI가 생성한 분석 결과 검토
- 실제 AWS 콘솔에서 확인
- 팀과 공유할 내용 정리

### 💡 실제 결과 예시
```
분석 결과:

1. 인스턴스 타입별 개수
   - t3.medium: 5대
   - t3.large: 3대
   - m5.xlarge: 2대

2. 실행 중: 8대, Stopped: 2대

3. ⚠️ Stopped 상태 인스턴스 (7일 이상)
   - i-0abc123 (t3.large) - 비용 낭비 중
   - i-0def456 (m5.xlarge) - 확인 필요

4. 🏷️ 이름 태그 없음: 3대
   - 관리 어려움, 태그 정책 위반

5. 최적화 제안
   - Stopped 인스턴스 종료 검토
   - 태그 정책 적용
   - t3.medium → t3.small 다운사이징 검토
```

## ⚡ Quick Win #2: 에러 로그 분석 (10분)

### 상황
CloudWatch 로그에 에러가 많이 발생했는데, 패턴을 찾기 어려움

### 단계별 가이드

**1단계: 로그 수집 (2분)**
```bash
aws --profile prod logs filter-log-events \
  --log-group-name /aws/lambda/api-handler \
  --start-time $(date -d '1 hour ago' +%s)000 \
  --filter-pattern "ERROR" \
  --max-items 100 > errors.log
```

**2단계: AI 분석 요청 (1분)**
```
이 에러 로그를 분석해서:

[errors.log 내용 붙여넣기]

1. 가장 많이 발생한 에러 TOP 3
2. 에러 발생 시간대 패턴
3. 각 에러의 가능한 원인
4. 우선순위별 해결 방법
5. 모니터링 개선 제안
```

**3단계: 액션 (7분)**
- 가장 빈번한 에러부터 해결
- 알람 설정 개선
- 팀에 공유

### 💡 실제 결과 예시
```
에러 분석 결과:

1. TOP 3 에러
   ① TimeoutError: 45회 (60%)
   ② ConnectionError: 20회 (27%)
   ③ ValidationError: 10회 (13%)

2. 시간대 패턴
   - 14:00-15:00: 집중 발생 (점심시간 이후 트래픽 증가)
   - 주로 평일 오후

3. 원인 분석
   ① TimeoutError
      - RDS 연결 타임아웃 (3초)
      - 가능 원인: DB 부하, 네트워크 지연
   
   ② ConnectionError
      - 외부 API 호출 실패
      - 가능 원인: API 서버 불안정

4. 해결 방법 (우선순위)
   🔴 긴급: RDS 성능 확인 및 타임아웃 증가
   🟡 중요: 외부 API 재시도 로직 추가
   🟢 개선: 입력 검증 강화

5. 모니터링 개선
   - TimeoutError 발생 시 즉시 알람
   - DB 성능 메트릭 대시보드 추가
   - 외부 API 헬스체크 자동화
```

## ⚡ Quick Win #3: RCA 문서 작성 (10분)

### 상황
장애가 발생했고, RCA 문서를 빠르게 작성해야 함

### 단계별 가이드

**1단계: 정보 정리 (3분)**
```
발생 시간: 2025-01-15 14:30 KST
증상: API Gateway 5xx 에러 급증 (정상 0.1% → 15%)
영향: 고객사 A 프로덕션 환경, 약 1,000명 사용자
감지: CloudWatch 알람
조치: Lambda 동시 실행 제한 증가 (100 → 500)
결과: 14:45 정상화 (15분 소요)
```

**2단계: AI에게 RCA 작성 요청 (2분)**
```
당신은 베스핀글로벌의 시니어 클라우드 엔지니어입니다.
다음 장애 정보로 RCA 문서를 작성해주세요:

[위 정보 붙여넣기]

다음 형식으로:
1. Executive Summary (경영진용, 3줄)
2. Timeline (시간순 상세)
3. Root Cause Analysis
4. Impact Assessment
5. Immediate Actions Taken
6. Preventive Measures (재발 방지)
7. Action Items (담당자, 기한)
```

**3단계: 검토 및 배포 (5분)**
- 생성된 문서 검토
- 실제 상황과 맞는지 확인
- 필요시 수정
- Confluence/Jira에 업로드

### 💡 실제 결과 예시
```markdown
# RCA: API Gateway 5xx Error Spike

## Executive Summary
2025-01-15 14:30, 고객사 A 프로덕션 환경에서 Lambda 동시 실행 제한 
도달로 인한 API 장애 발생. 15분 내 제한 증가로 해결. 재발 방지를 위해 
Auto Scaling 정책 및 모니터링 강화 필요.

## Timeline
| 시간 | 이벤트 | 담당자 |
|------|--------|--------|
| 14:30 | CloudWatch 알람 발생 | System |
| 14:32 | 온콜 엔지니어 확인 시작 | 김OO |
| 14:35 | Lambda 제한 도달 확인 | 김OO |
| 14:38 | 제한 증가 요청 승인 | 박OO |
| 14:40 | 제한 100→500 증가 적용 | 김OO |
| 14:45 | 에러율 정상화 확인 | 김OO |
| 14:50 | 고객사 커뮤니케이션 | 이OO |

## Root Cause
Lambda 함수의 동시 실행 제한(100)이 예상보다 빠르게 도달.
점심시간 이후 트래픽 급증 패턴을 사전에 파악하지 못함.

## Impact Assessment
- 영향 사용자: 약 1,000명
- 장애 시간: 15분
- 에러율: 15% (약 150건 실패)
- 비즈니스 영향: 중간 (주문 처리 지연)

## Immediate Actions
✅ Lambda 동시 실행 제한 증가 (100 → 500)
✅ 고객사 커뮤니케이션 완료
✅ 모니터링 강화

## Preventive Measures
1. Auto Scaling 정책 수립
   - 동시 실행 70% 도달 시 자동 증가
   - 최대 1,000까지 확장 가능하도록 설정

2. 모니터링 개선
   - 동시 실행 60% 도달 시 Warning 알람
   - 트래픽 패턴 분석 대시보드 구축

3. 용량 계획
   - 주간 트래픽 리뷰 프로세스 수립
   - 이벤트 전 사전 스케일링

## Action Items
| 항목 | 담당자 | 기한 | 상태 |
|------|--------|------|------|
| Auto Scaling 정책 적용 | 김OO | 01/17 | 진행중 |
| 모니터링 대시보드 구축 | 박OO | 01/20 | 예정 |

| 고객사 사후 보고서 발송 | 최OO | 01/16 | 완료 |
```
